> **Requires:** A keccak256-capable execution environment (EVM canonical; non-EVM

> **Supersedes:** None

## Abstract

TAS-1 is an open standard for **cryptographic attribution of work in a
permissionless model training run**. This v1.0 covers compute (gradient)
contributions; data and capital attribution are anticipated as v1.x minor
extensions using the same primitives. The standard specifies (1) a canonical
byte format for hashing a gradient tensor, (2) two complementary Merkle-leaf
encodings —
an off-chain *Lab Commitment* binding contributor identity to a specific
gradient artifact, and an on-chain *Contribution Record* binding contributor
identity to a protocol-defined contribution weight, (3) a minimal chain
interface for anchoring commitment roots and redeeming inclusion proofs, and
(4) replay-protection semantics that prevent any leaf from being credited
twice.

TAS-1 does **not** specify how a gradient is produced, how a contribution
weight is derived from a gradient, how a token is issued against a
contribution, how a model is hosted, or how inference is monetized. These
are concerns of the implementing protocol. TAS-1 specifies only the bytes
on the wire and the leaf semantics.

The intent is that any decentralized training protocol — Bittensor subnet,
Gensyn-style compute marketplace, standalone EVM protocol, or non-EVM L1 —
can implement TAS-1 and produce attribution receipts that any other TAS-1
implementation can verify.

## Motivation

Decentralized model training has produced its first credible empirical
result — Templar's Covenant-72B, March 2026, 67.1 MMLU at fully
permissionless scale. But the projects building in this space share a gap:
**there is no portable standard for crediting the contributors who produced
the model.** Templar uses mutable Cloudflare R2 blobs. Macrocosmos IOTA's
attribution scheme is roadmap-only. Pluralis Protocol Models hide weights
but don't formalize attribution. Gensyn's checkpoint-replay scheme inherits
known spoofability concerns. Each project's attribution layer is internal
plumbing, not a portable artifact.

Without a standard, every new protocol re-invents:
- A gradient identity scheme
- A contributor-to-gradient binding
- An on-chain attribution record
- Replay protection

And without a standard, *no* protocol can credibly claim that a contributor's
work in protocol A is the same artifact as the work claimed in protocol B,
or that an inference-revenue split derived from one run is reproducible by
an external auditor.

TAS-1 formalizes the smallest possible surface that solves these problems:
**given a gradient artifact and a contributor address, produce a pair of
Merkle leaves that any verifier — on-chain contract, off-chain auditor,
zk-circuit, or fraud-proof challenger — can independently reconstruct.**

The standard is intentionally agnostic to:
- Whether the model is open or closed
- Whether the contribution weight is derived from loss-delta scoring,
  validator quorum, ZK-proven training, or any other scheme
- Whether monetization happens through token issuance, inference revenue
  split, fee-sharing, or no monetization at all
- Whether the chain anchoring the commitment is EVM, Substrate, Solana,
  Cosmos SDK, or a future architecture

TAS-1 says only: **here are the bytes; here is the leaf format; here is the
replay rule.**

## Terminology

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**,
**SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, and **OPTIONAL** in
this document are to be interpreted as described in RFC 2119.

- **Run.** A bounded training process with a defined start time, a defined
  contribution window end, and a unique run identifier `runId` (32 bytes).
- **Lab.** The party that initiates a run, defines its evaluation procedure,
  and publishes per-cycle commitments. The lab is the trust anchor of a
  TAS-1 v1 run; TAS-2+ extensions reduce or eliminate this trust.
- **Contributor.** Any entity that submits work — gradients, data, capital,
  validation effort — to a run. Identified by a 20-byte address (EVM
  canonical).
- **Sync Cycle.** A discrete unit of time within a run. Each cycle has a
  monotonically increasing integer index `syncCycle` ≥ 1.
- **Gradient.** A real-valued tensor of arbitrary rank and shape submitted
  by a contributor as their work product for a sync cycle. TAS-1 hashes
  gradients; it does not interpret their semantics.
- **Gradient Hash.** The canonical 32-byte keccak256 hash of a gradient
  tensor under §4.1.
- **Contribution Weight.** A non-negative integer ≥ 1 representing the
  protocol-defined value of a contributor's work in a sync cycle. The
  derivation of weight from gradient (or other contribution) is OUT OF
  SCOPE for TAS-1.
- **Contribution Type.** A `uint8` (0–255, with the standard reserving
  0–2 for compute / data / capital and 3–255 implementation-defined).
- **Lab Commitment.** A Merkle root over §4.2 leaves, published per
  (run, sync_cycle), that anchors which gradients the lab accepted in
  that cycle.
- **Contribution Record.** A Merkle leaf under §4.3 that an on-chain
  registry consumes to credit a contributor with weight in a run.

## Specification

### 4.1 Canonical Gradient Hash

A gradient is a real-valued tensor of arbitrary rank and shape. To hash
a gradient under TAS-1, the implementation MUST:

1. Reject any tensor whose dtype is not a real-valued floating-point
   type (integer and complex tensors are not gradients under this
   standard).
2. Convert the tensor to **IEEE 754 binary32 (single-precision)**. If
   the input was already binary32, conversion is a no-op. If the input
   was binary64, binary16, bfloat16, or any other real-valued float
   type, the implementation MUST cast to binary32 with round-to-nearest-
   even rounding before serialization.
3. Ensure the tensor is in **C-contiguous (row-major) memory order**.
4. Construct the canonical serialization:
   ```
   header  := u32_le(rank) || u32_le(dim_0) || u32_le(dim_1) || ... || u32_le(dim_{rank-1})
   payload := tensor bytes, IEEE 754 binary32 little-endian, row-major
   serialized := header || payload
   ```
   where `u32_le` denotes 4-byte little-endian unsigned integer encoding.
5. Compute `gradient_hash := keccak256(serialized)`. The result is a
   32-byte digest.

A scalar gradient (rank 0) MUST be encoded with `header = u32_le(0)` and
`payload` equal to the single binary32 little-endian value (4 bytes).

A gradient with any zero-length dimension MUST be encoded with the rank
and dimension headers as above, and `payload` equal to zero bytes. Such
tensors are valid TAS-1 gradients.

#### 4.1.1 Canonicalization rationale

The fp32 cast is mandatory and non-negotiable. Distributed training
varies in dtype across contributors — fp16 on consumer GPUs, bf16 on
TPUs and H100, fp32 on CPU, fp8 on Blackwell. Hashing in the
contributor's native dtype would produce different gradient hashes for
the same logical gradient depending on the contributor's hardware,
making cross-contributor verification impossible. fp32 is the canonical
upcast: every modern accelerator can produce it deterministically, and
the round-trip rounding error from native compute → fp32 → hash is
acceptable for content addressing under v1 of this standard.

TAS-2 extensions MAY relax this requirement in favor of TEE-attested
or ZK-witness commitments to the original-precision tensor. The
present specification does not depend on those extensions.

#### 4.1.2 Hash agility

TAS-1 fixes keccak256 as the gradient-hash primitive. The 32-byte
output is byte-aligned with EVM `bytes32` and the standard's other
leaf encodings.

Future revisions MAY introduce alternate hash primitives (BLAKE3,
Poseidon, etc.) under different version numbers (TAS-1.1, TAS-2). A
single run MUST use exactly one hash primitive end-to-end; mixing is
not permitted.

### 4.2 Layer 1: Lab Commitment Leaf

A Lab Commitment Leaf binds a contributor address, a sync cycle, and a
gradient hash to a single 32-byte digest. The lab MUST construct one
leaf per accepted gradient in a sync cycle, build a Merkle tree over
all accepted leaves under §4.4, and publish the root.

**Encoding:**

```
leaf_L1 := keccak256(
    addr_bytes(20) || cycle_bytes(32) || gradient_hash(32)
)
```

where:
- `addr_bytes` is the contributor's 20-byte address (EVM canonical),
  case-folded to lowercase before hashing.
- `cycle_bytes` is the sync cycle index encoded as **32-byte big-endian
  unsigned integer**.
- `gradient_hash` is the 32-byte digest from §4.1.

The total preimage length is 84 bytes.

#### 4.2.1 Why three fields, no more

The Layer 1 leaf binds the **smallest tuple that proves a gradient was
submitted by a specific contributor in a specific cycle**. Adding more
fields (timestamps, metadata, run identifiers) is rejected because:

- The run identifier is implicit: a Lab Commitment Root is published
  per `(runId, syncCycle)`, so the root's location in the chain
  protocol already binds the run.
- Timestamps are non-deterministic across contributors and would
  prevent identical gradients from producing identical leaves.
- Metadata (model version, batch index, etc.) is the lab's
  responsibility to publish through other channels and does not need
  cryptographic binding under this standard.

#### 4.2.2 What Layer 1 proves

A Layer 1 inclusion proof for `(contributor, syncCycle, gradient_hash)`
proves: *"the lab accepted gradient X from contributor Y in sync cycle
C of run R."*

It does NOT prove:
- That the gradient improved the model
- That the gradient was honestly computed (rather than adversarially
  crafted to pass the lab's check)
- That the lab itself behaved honestly

These guarantees require TAS-2+ extensions (multi-validator quorum,
fraud proofs, ZK proofs of training).

### 4.3 Layer 2: Contribution Record Leaf

A Contribution Record Leaf binds a contributor address, a contribution
weight, a contribution type, and a sync cycle to a single 32-byte
digest. The on-chain Registry MUST consume one Layer 2 leaf per
contributor-cycle pair, decrementing only on the redeeming inclusion
proof's first valid call.

**Encoding (EVM canonical):**

```solidity
leaf_L2 := keccak256(
    bytes.concat(
        keccak256(
            abi.encode(contributor, weight, contributionType, syncCycle)
        )
    )
)
```

where `abi.encode` is the Solidity ABI v2 encoding with each field
zero-padded to 32 bytes:

| Field             | Solidity type | Encoded width | Notes                          |
|-------------------|---------------|---------------|--------------------------------|
| `contributor`     | `address`     | 32 bytes      | left-padded with zeros         |
| `weight`          | `uint256`     | 32 bytes      | big-endian                     |
| `contributionType`| `uint8`       | 32 bytes      | left-padded with zeros         |
| `syncCycle`       | `uint256`     | 32 bytes      | big-endian                     |

Total preimage length to the inner `keccak256` is 128 bytes.

The **outer-keccak-of-inner-keccak** wrapping is the OpenZeppelin
canonical leaf format and prevents second-preimage attacks at the
boundary between leaves and intermediate Merkle nodes.

#### 4.3.1 Reserved contribution types

| Type | Meaning         |
|------|-----------------|
| 0    | Compute         |
| 1    | Data            |
| 2    | Capital         |
| 3–255| Implementation-defined |

Implementations MAY define additional types ≥ 3. Implementations MUST
NOT reassign types 0–2.

#### 4.3.2 Non-EVM implementations

Implementations on chains without Solidity ABI encoding MUST produce a
byte-identical preimage to `abi.encode(address, uint256, uint8,
uint256)` as defined above. The simplest correct implementation is to
construct the 128-byte buffer directly:

```
preimage := zeros(12) || addr_bytes(20) || u256_be(weight) || zeros(31) || u8(type) || u256_be(cycle)
inner   := keccak256(preimage)
leaf_L2 := keccak256(inner)
```

### 4.4 Merkle Tree Construction

TAS-1 uses **OpenZeppelin commutative pair-hashing** for both Layer 1
and Layer 2 trees.

#### 4.4.1 Pair hashing

For two 32-byte values `a` and `b`:

```
hash_pair(a, b) := if a < b then keccak256(a || b) else keccak256(b || a)
```

where `<` is byte-wise lexicographic comparison.

#### 4.4.2 Tree construction

Given a non-empty list of leaves `[L_0, L_1, ..., L_{n-1}]`:

1. The tree's bottom layer is the leaf list itself.
2. To produce the next layer up, pair each adjacent pair of nodes and
   apply `hash_pair`. If the layer has an odd number of nodes, the last
   node is **promoted unchanged** to the next layer (it is not paired
   with itself).
3. Repeat until the layer contains a single node. That node is the
   Merkle root.

A tree built over `n = 0` leaves is undefined under this standard;
implementations MUST reject empty trees.

A tree built over `n = 1` leaf has root equal to the leaf itself.

#### 4.4.3 Inclusion proof

An inclusion proof for leaf `L_i` is the ordered list of sibling
hashes encountered during the leaf-to-root walk. To verify:

```
h := L_i
for sibling in proof:
    h := hash_pair(h, sibling)
return h == claimed_root
```

The sibling order is the order siblings are encountered moving up the
tree, with the proof empty for a single-leaf tree.

The on-chain reference is OpenZeppelin's `MerkleProof.verifyCalldata`.

### 4.5 Required Chain Interface

A TAS-1 conformant chain implementation MUST expose at least the
following surface. Function names MAY differ; semantics MUST NOT.

```solidity
interface ITAS1Registry {
    /// Register a training run. Implementations MAY add fields
    /// (tier, anchor lab, conversion rates, etc.) but MUST preserve
    /// the (runId, registerer, contributionWindowEnd, closed) state.
    event TrainingRunRegistered(
        bytes32 indexed runId,
        address indexed registerer,
        uint256 contributionWindowEnd
    );

    /// Anchor the Lab Commitment / Contribution Aggregate root for a
    /// (run, sync_cycle). MUST reject:
    ///   - duplicate (runId, syncCycle) submissions
    ///   - submissions after run.contributionWindowEnd
    ///   - submissions to a closed run
    ///   - submissions from any party other than the run registerer
    ///     (or a delegated framework address; implementation choice)
    function submitProofBundle(
        bytes32 runId,
        bytes32 merkleRoot,
        uint256 totalNodes,
        uint256 syncCycle
    ) external;

    event ProofBundleSubmitted(
        bytes32 indexed runId,
        uint256 indexed syncCycle,
        bytes32 merkleRoot,
        uint256 totalNodes
    );

    /// Redeem a Layer 2 inclusion proof and credit `weight` to
    /// `contributor` under `runId`. MUST:
    ///   - reconstruct the Layer 2 leaf per §4.3
    ///   - verify the leaf's inclusion in proofBundleRoots[runId][syncCycle]
    ///     under §4.4.3
    ///   - reject if the leaf has already been consumed
    ///   - mark the leaf consumed atomically with the credit
    function recordContribution(
        bytes32 runId,
        address contributor,
        uint256 weight,
        uint8 contributionType,
        uint256 syncCycle,
        bytes32[] calldata merkleProof
    ) external;

    event ContributionRecorded(
        bytes32 indexed runId,
        address indexed contributor,
        uint256 weight,
        uint8 contributionType,
        uint256 syncCycle
    );
}
```

#### 4.5.1 Run lifecycle

A run progresses through three phases:

1. **Open** — `submitProofBundle` and `recordContribution` are both
   accepted. The window is bounded by `contributionWindowEnd`.
2. **Window-closed but run-open** — `submitProofBundle` is rejected;
   `recordContribution` is still accepted (contributors can redeem
   inclusion proofs against previously-anchored roots).
3. **Closed** — neither call is accepted. Implementations MUST provide
   a `closeTrainingRun(bytes32 runId)` (or equivalent) that transitions
   to the closed state and triggers any post-run accounting (token
   issuance, fee distribution, etc., out of scope for TAS-1).

#### 4.5.2 Replay protection

The Registry MUST maintain a per-run set of consumed leaves
(`consumedLeaves[runId][leaf] -> bool`). `recordContribution` MUST
revert if the reconstructed leaf is already in the set. This prevents:

- Double-credit if the same leaf is submitted twice
- Cross-cycle replay (the `syncCycle` field in the leaf encoding
  binds the leaf to one cycle)
- Cross-run replay (consumedLeaves is keyed by runId)

#### 4.5.3 Aggregator vs. direct-redemption flow

Implementations MAY allow either:
- **Direct redemption.** Contributors call `recordContribution`
  themselves with their own inclusion proof.
- **Aggregator-mediated redemption.** A trusted aggregator collects
  contributions, builds the Merkle tree, anchors the root, and either
  serves proofs to contributors (who then call `recordContribution`)
  or calls `recordContribution` on contributors' behalf.

Both modes are TAS-1 conformant. Aggregator-mediated redemption is
RECOMMENDED for v1 implementations because it amortizes per-cycle
gas costs and centralizes proof storage.

### 4.6 Recommended Off-Chain Checks

A conformant aggregator or sequencer SHOULD enforce the following at
the contribution-acceptance boundary:

#### 4.6.1 Contributor signature (RECOMMENDED)

Each contribution submitted to the aggregator SHOULD be accompanied by
an EIP-191 signature over the canonical contribution message, signed
by the contributor's private key:

```
message := keccak256(
    "TAS-1 contribution v1\n" ||
    runId(32) || contributor(20) || syncCycle(32_be) ||
    weight(32_be) || contributionType(1) || gradient_hash(32)
)
signature := EIP-191 sign(message)
```

The aggregator SHOULD verify `ecrecover(message, signature) ==
contributor` before accepting the contribution. This prevents
weight-forgery attacks where a compromised aggregator API key is
used to submit fabricated contributions in another contributor's name.

The on-chain `recordContribution` does NOT verify this signature; it
verifies only the Merkle inclusion proof against the on-chain root.
The signature check is an off-chain integrity layer that gates what
the aggregator commits to.

#### 4.6.2 Lab commitment precondition (REQUIRED)

An aggregator MUST reject any contribution whose
`(runId, syncCycle, gradient_hash)` tuple does not appear in the
lab's previously-published Layer 1 commitment for that cycle, verified
via Merkle inclusion proof per §4.4.3.

#### 4.6.3 Loss-improvement threshold (IMPLEMENTATION-DEFINED)

The lab SHOULD apply a minimum loss-improvement threshold before
accepting a gradient into a Layer 1 commitment. A reference threshold
table for v1 single-lab runs:

| Tier | Description                       | Default threshold |
|------|-----------------------------------|-------------------|
| 1    | Frontier (≥ 70B params)           | 1 bp (0.0001)     |
| 2    | Mid-scale (≥ 1B, < 70B)           | 0.5 bp (0.00005)  |
| 3    | Small (< 1B)                      | 0.1 bp (0.00001)  |

These are recommendations for single-lab v1 implementations. Multi-
validator v2 implementations SHOULD derive thresholds from quorum
median rather than a fixed table.

The threshold semantics are:

```
relative_improvement := (baseline_loss - candidate_loss) / baseline_loss
accept iff candidate_loss < baseline_loss AND relative_improvement >= threshold
```

## Conformance Levels

A TAS-1 implementation MAY declare conformance at one of two levels.
**Level B is RECOMMENDED for new implementations.**

### Level A — Basic Attribution

An implementation conforms at Level A if and only if it:

- Implements §4.1 (canonical gradient hash) byte-for-byte
- Implements §4.2 (Layer 1 leaf) byte-for-byte
- Implements §4.3 (Layer 2 leaf) byte-for-byte
- Implements §4.4 (Merkle tree construction) using OpenZeppelin
  commutative pair-hashing
- Exposes a chain interface satisfying §4.5
- Enforces §4.6.2 (lab commitment precondition) at the aggregator
  boundary
- Maintains replay protection per §4.5.2
- Passes the conformance test vectors

Off-chain integrity (contributor signatures, threshold gating) is
the implementer's choice at Level A. A Level A implementation
provides cryptographic attribution but does not cryptographically
bind the contributor to their submission at the aggregator
boundary.

### Level B — Signed Attribution

An implementation conforms at Level B if and only if it:

- Conforms at Level A
- Treats §4.6.1 (EIP-191 contributor signatures) as REQUIRED rather
  than RECOMMENDED at the aggregator boundary
- Rejects any contribution whose
  `ecrecover(message, signature) != contributor`

A Level B implementation cryptographically prevents weight forgery
even if the aggregator's API key is compromised. New
implementations SHOULD target Level B; existing implementations
MAY adopt Level A first and upgrade.

### Conformance Declaration

Implementations SHOULD publicly declare their conformance level via
at least one of:

- A Solidity constant:
  `string public constant TAS_VERSION = "TAS-1 v1.0 Level B";`
- An HTTP header on the aggregator API:
  `X-TAS-Conformance: TAS-1 v1.0 Level B`
- A field in the run-registration payload that contributors can
  inspect before submitting work

Users SHOULD be able to verify a protocol's claimed conformance
level by running the TAS-1 conformance suite against the
implementation. Mismatched declarations are a protocol-violation
issue.

## Security Considerations

### What TAS-1 v1 guarantees

1. **Gradient identity.** Two contributors who submit byte-identical
   gradients (after fp32 canonicalization) produce identical
   `gradient_hash` values; any byte-level difference produces
   different hashes with overwhelming probability (keccak256
   collision-resistance).
2. **Contributor binding at the lab.** A Layer 1 inclusion proof is
   sufficient evidence that the lab committed to a specific
   `(contributor, cycle, gradient_hash)` tuple at a known root. An
   adversary cannot forge contributor identity without ecrecover-
   colliding their address (cryptographically infeasible) **AND**
   passing the lab's acceptance check.
3. **On-chain attribution unforgeability.** A Layer 2 inclusion proof
   redeemed via `recordContribution` is verified by the chain against
   a previously-anchored root. The chain itself is the trust anchor
   for what was anchored when.
4. **Replay protection.** `consumedLeaves` ensures a leaf is credited
   at most once per run.

### What TAS-1 v1 does NOT guarantee

1. **Gradient quality.** TAS-1 does not specify how a lab decides which
   gradients to accept. A malicious lab can accept noise gradients;
   a malicious contributor can submit an adversarial gradient that
   passes a loss-delta check while degrading the model on out-of-
   distribution inputs (Fang/Jia/Dziedzic/Papernot, arxiv 2208.03567).
2. **Computation honesty.** TAS-1 does not prove that the gradient was
   honestly computed by the contributor. Replay, collusion, and
   "shadow-trained" gradients are all consistent with a valid TAS-1
   attribution receipt.
3. **Lab honesty.** The lab decides what enters the Layer 1
   commitment. A malicious lab can refuse to commit honest
   contributions, fabricate commitments crediting accomplices, or
   selectively reorder the cycle to disadvantage specific
   contributors.

These gaps are addressed by TAS-2+ extensions (multi-validator
quorum, optimistic fraud proofs, ZK proofs of training, TEE
attestation). v1 single-lab implementations MUST clearly document
their trust assumptions to contributors.

### Cross-implementation byte-equivalence

A non-EVM implementation that produces leaves matching §4.2 / §4.3
exactly (byte-for-byte preimages and keccak256 outputs) is TAS-1
conformant regardless of the underlying chain. This enables:

- Cross-chain attribution receipts (a contribution recorded on
  Solana can be verified by an Ethereum auditor via standard zk-
  bridge proofs, since the leaf format is identical)
- Off-chain audit tooling that does not depend on a specific chain
- Migration from one chain to another without reconstructing
  attribution records

Implementations MUST use the OpenZeppelin commutative pair-hash
ordering. Implementations MUST NOT introduce position-dependent leaf
encodings (no left-vs-right byte tagging) since this would break
cross-implementation byte-equivalence.

### Hash agility and quantum considerations

keccak256 is the canonical TAS-1 hash for the foreseeable future.
Post-quantum migration, if needed, will be handled by a future TAS
revision (TAS-2 or later) under a new spec number, not by an in-place
upgrade of this document.

Implementations MUST NOT mix hash functions within a single run.

### Time-of-check / time-of-use at the aggregator

The recommended off-chain flow under §4.6 has a TOCTOU window: between
the aggregator verifying a contributor's signature and the on-chain
`recordContribution` reverting on duplicate consumption. Implementations
SHOULD ensure idempotency at the aggregator (per §4.6.2 and the
reference implementation's `PendingContribution` deduplication) so that
a contributor retrying a stalled submission does not produce divergent
state.

### Front-running of `recordContribution`

`recordContribution` is permissionless — anyone with a valid inclusion
proof can call it on behalf of any contributor. An adversary front-
running a contributor's transaction simply records the contribution
to the contributor's address (since the contributor address is
embedded in the leaf). The adversary pays gas; the contributor
receives credit. This is a feature, not a bug: it enables aggregator-
sponsored redemption (§4.5.3) without explicit permissioning.

## Extension Hooks for Future Versions

The v1 specification deliberately leaves the following surfaces
extensible:

### Validator quorum (TAS-2 candidate)

A future version is expected to specify:
- `registerValidator(uint256 stake)` and validator state
- `submitLossReport(runId, syncCycle, contributor, lossDelta, signature)`
- A median-aggregation function from per-validator loss reports to
  contribution weight
- Bond-erosion (or slashing) for validators whose reports diverge
  from the quorum median

The Layer 1 / Layer 2 leaf formats and chain interface defined in this
document are forward-compatible: v2 weights become deterministic
quorum outputs rather than lab-declared integers, but the on-chain
`recordContribution` signature does not change.

### Fraud-proof challenges (TAS-3 candidate)

A future version is expected to specify:
- A challenge window opening on each `submitProofBundle`
- A bisection game over the gradient computation graph (TAO-style;
  arxiv 2510.16028)
- Per-operator IEEE-754 + empirical-percentile tolerance bounds
- Slashing of the proposer on successful challenge

The challenge mechanism will require new contracts (`ChallengeManager`)
and a witness-data-availability commitment, but does not modify the
TAS-1 leaf formats or core attribution interface.

### ZK audit checkpoints (TAS-4 candidate)

A future version is expected to specify:
- Periodic Kaizen / zkLoRA-style proofs that a model's training
  trajectory is verifiable end-to-end
- Anchoring of zk-proof commitments alongside `submitProofBundle`
  roots

These extensions are on the TAS Working Group's roadmap and will be
specified in subsequent TAS documents.

## Versioning

TAS-1 follows a strict major.minor versioning scheme:

- **Major version** changes (TAS-1 → TAS-2) introduce non-backward-
  compatible changes to leaf formats, hash primitives, or the chain
  interface.
- **Minor version** changes (v1.0 → v1.1) clarify ambiguous text, add
  test vectors, or extend recommended off-chain checks. Minor versions
  MUST be byte-equivalent at the leaf-format level.

Implementations SHOULD declare their conformance level as
`"TAS-1 v1.0"` in their public interface (e.g., as a Solidity
constant or HTTP header).
